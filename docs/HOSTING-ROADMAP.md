# MathBoard — Hosting & Domain Roadmap (go-live architecture)

**Goal:** run MathBoard in production for the lowest possible recurring cost, on a
portal that is easy to manage from one dashboard, with glitch-free behaviour on
every device (iPad Safari, Android Chrome, desktop, low-end hardware).

**Decision: stay on Cloudflare.** The app is already architected as a Cloudflare
Worker (static assets + gated `/api/*` and `/content/*`) plus a collab Worker with
Durable Objects. Cloudflare is simultaneously the cheapest and the most
user-friendly option here because hosting, domain, DNS, SSL, CDN, and analytics
all live in **one dashboard** — nothing to glue together, nothing extra to pay for.

## 1. Cost breakdown (what "very affordable" means concretely)

| Item | Provider | Price | Notes |
|---|---|---|---|
| Hosting (app + API + collab) | Cloudflare Workers **Paid** | **$5 / month** | Covers both Workers, Durable Objects, Vectorize, static assets, unlimited bandwidth on assets. Free plan can work at small scale, but $5 removes daily request caps and DO surprises. |
| Domain (e.g. `.com`) | Cloudflare **Registrar** | **≈ $11 / year, at cost** | Cloudflare sells domains at wholesale price — no markup, no renewal price jump, no upsell screens. Free WHOIS privacy included. |
| SSL / HTTPS | Cloudflare | **$0** | Automatic, auto-renewing. |
| CDN / global edge | Cloudflare | **$0** | Included; assets served from 300+ locations → fast on any device anywhere. |
| Auth + profiles DB | Supabase | **$0** (Free tier) | 50k monthly active users, 500 MB DB. Upgrade to Pro ($25/mo) only if you outgrow it. |
| Workers AI + Vectorize (RAG) | Cloudflare | ~$0 at launch | Generous included allotment on the $5 plan; pay-as-you-go beyond it. |

**Total: ≈ $5–6 / month + ~$11 / year.** No servers to patch, no OS updates,
nothing that can fall over at 2 AM.

**Domain portal choice:** buy the domain **inside the Cloudflare dashboard**
(Domain Registration → Register Domain). Because the site is hosted on Cloudflare
Workers, DNS records are created automatically when you attach the domain to the
Worker — zero manual DNS editing. If you prefer a separate registrar, Namecheap or
Porkbun (~$10–12/yr) also work: buy there, then point nameservers at Cloudflare
(two fields to paste, one time). Cloudflare Registrar remains the recommendation
because it is one login, at-cost pricing, and no renewal traps.

## 2. Why this runs glitch-free on any device

These properties already exist in the codebase — the roadmap protects them:

- **No build step, vanilla ES modules** — nothing to break between dev and prod.
- **Offline-first PWA** (`sw.js`, `manifest.json`) — works on flaky school Wi-Fi;
  after first load the whiteboard opens instantly even offline.
- **All libraries vendored in `vendor/`** — no CDN outage can ever break the app.
- **Edge caching via `_headers`** — hashed/versioned assets cached hard, `sw.js`
  and HTML always revalidated, so updates roll out immediately with no stale-cache
  glitches.
- **Ink pipeline** (snapshot-blit + Path2D cache in `js/app.js`) — pencil latency
  is the #1 acceptance criterion; nothing in deployment may regress it.
- **Cross-device QA gate** — `docs/BROWSER-TEST-GUIDE.md` must pass on iPad
  Safari, Android Chrome, and desktop before every deploy that touches
  `js/app.js`, `sw.js`, or `css/app.css`.

## 3. Phased roadmap

### Phase 0 — Buy domain & attach (½ day)
1. Cloudflare dashboard → **Domain Registration** → register the domain (~$11/yr).
2. Workers & Pages → `mathboard` → **Settings → Domains & Routes → Add custom
   domain** → enter the apex (and `www`). SSL + DNS are automatic.
3. Same for `mathboard-collab` on a subdomain, e.g. `collab.<domain>`.
4. Update `config.js` collab URL to the new subdomain; bump cache version.

### Phase 1 — Production hardening (1 day) ← *files generated by `CURSOR-HOSTING-TASK.md`*
- Custom-domain `routes` in both `wrangler.jsonc` files; disable `workers.dev`
  URL once the domain is live.
- `/api/health` endpoint + free uptime monitor (UptimeRobot / Cloudflare Health
  Checks) pinging it every 5 min.
- **Supabase keep-alive cron** in the Worker — free-tier Supabase projects pause
  after ~1 week of inactivity; a scheduled ping prevents users ever hitting a
  paused-database login failure.
- `scripts/go-live-check.mjs` preflight: cache-version match between
  `index.html` and `sw.js`, no secrets in tracked files, wrangler auth OK.
- Set secrets (`SUPABASE_JWT_SECRET`, `INGEST_TOKEN`) via `wrangler secret put`.

### Phase 2 — Launch QA (1–2 days)
- Finish RAG ingest batches (`content/INGEST-PROGRESS.md`).
- Run the full `docs/BROWSER-TEST-GUIDE.md` matrix on real devices, on the real
  domain (service worker + PWA install behave differently than on localhost).
- Verify offline mode: load app, airplane mode, reopen — whiteboard must work.
- Lighthouse pass on mobile: PWA installable, no console errors.

### Phase 3 — Post-launch guardrails (ongoing, ~1 hr/month)
- Cloudflare dashboard → Workers analytics: watch request counts vs. plan
  allotment (alerts are built in).
- Supabase dashboard → Database size + MAU vs. free-tier limits; upgrade to Pro
  ($25/mo) only when >~40k MAU or >400 MB DB.
- Keep the deploy ritual: bump `?v=NN` + `sw.js` cache name → `scripts/go-live-check.mjs`
  → `scripts/deploy-mathboard.sh` → spot-check on one phone + one iPad.

### Cleanup (optional, low risk)
`netlify.toml` and `vercel.json` are leftovers from earlier hosting experiments;
`_headers` is the live source of truth on Cloudflare. Keep them only if you want
a fallback host documented; otherwise delete to avoid confusion.

to 100k on the same $5 plan plus usage.
